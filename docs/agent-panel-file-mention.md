# Agent 面板 `@ 文件` 功能交互文档

本文档仅描述当前已实现的功能交互与用户可感知行为，不展开技术实现细节。

## 1. 功能定义

Agent 面板输入框支持通过 `@` 引用当前工作区中的文件或文件夹。

用户在输入框中输入 `@` 并检索目标后，可将目标插入为一个不可拆分的引用 token。该 token 在输入框内以名称形式展示，在真正发送给 Agent 时会被序列化为带相对路径的引用文本。

## 2. 生效范围与前置条件

- 功能仅存在于 Agent 面板输入框内。
- 必须已打开工作区。未打开工作区时，输入框不可编辑，占位文案为 `Open a folder first.`，`@` 功能不可用。
- 候选来源为当前工作区树中的全部文件与文件夹，递归覆盖各级目录。
- 工作区根目录本身不会作为候选项出现；仅根目录下的节点及其后代节点可被引用。
- 候选范围与当前工作区树一致，因此会排除工作区树本身已忽略的内部目录或产物目录。
- 当前已排除的目录名包括：
  - `.git`
  - `.pi`
  - `.tmp-icon-theme-cache`
  - `dist`
  - `dist-electron`
  - `node_modules`
  - `temp-icon-theme-cache`

## 3. 触发规则

### 3.1 基本触发条件

只有同时满足以下条件时，`@` 才会进入“文件引用检索态”：

- 输入框已聚焦。
- 光标是插入点状态，不是范围选区。
- 光标位于普通文本中，不在已插入的引用 token 内部。
- 光标前方最近一段文本满足 `@查询词` 结构。

### 3.2 `@` 前置边界

`@` 只有在以下位置之一出现时才会触发：

- 文本开头。
- 空白字符之后。
- 以下字符之后：
  - `(`
  - `[`
  - `{`
  - `'`
  - `"`
  - `` ` ``

因此，普通邮箱或连续单词中的 `@` 不会触发候选。例如：

- `@read`：会触发。
- `请看 @read`：会触发。
- `hello@example.com`：不会触发。

### 3.3 查询词允许与终止条件

`@` 之后的查询词允许继续输入，直到遇到以下任一字符或状态时，当前查询结束：

- 空白字符。
- 第二个 `@`。
- `[` `]` `(` `)` `{` `}`。
- 光标移动到别处。
- 出现范围选区。
- 输入框失焦。

查询词支持：

- 中英文混输。
- 路径片段输入。
- `/` 或 `\` 形式的路径分隔。
- 大小写不敏感匹配。

## 4. 候选列表

### 4.1 打开时机

当输入框处于 `@查询词` 检索态时，候选列表显示在输入框上方。

### 4.2 候选内容

每个候选项包含：

- 文件或文件夹图标。
- 节点名称。
- 父级相对路径（若存在）。

显示规则：

- 一级节点只显示名称，不显示父级路径。
- 多级节点显示名称 + 父级相对路径，用于区分同名项。

### 4.3 覆盖对象

候选同时包含：

- 文件。
- 文件夹。

文件夹可被引用，行为与文件一致，只是路径对应文件夹的相对路径。

### 4.4 默认结果

当用户仅输入 `@`、尚未继续输入查询词时：

- 会展示候选列表。
- 文件夹整体优先于文件。
- 同类项按工作区树遍历顺序展示。

### 4.5 搜索与排序

输入查询词后，候选按相关度排序。当前优先级可概括为：

1. 名称、展示名称或完整相对路径的完全匹配。
2. 名称或展示名称前缀匹配。
3. 名称或展示名称包含匹配。
4. 相对路径前缀匹配。
5. 相对路径包含匹配。
6. 路径片段前缀匹配。
7. 多词同时命中同一项的名称或路径信息。

同分时：

- 文件夹优先于文件。
- 再按工作区树原始顺序稳定排序。

搜索特征：

- 匹配大小写不敏感。
- 会统一处理正斜杠与反斜杠。
- 可直接按文件名搜。
- 可按相对路径搜。
- 可按路径片段搜。
- 可用多词搜索，词之间可用空格或路径分隔符拆分。

例如，输入 `app` 时，会优先命中 `App.tsx`、`app-titlebar.tsx` 这类名称直接相关的项；输入 `src/components/app` 时，也可按路径命中。

### 4.6 结果数量与空状态

- 候选列表最多返回 100 条结果。
- 超出时列表可滚动。
- 无匹配结果时，列表仍保留，并显示空状态文案：`No matching files or folders`。

## 5. 候选列表交互

### 5.1 鼠标交互

- 鼠标移入候选项时，该项变为当前高亮项。
- 鼠标点击候选项时，立即插入该引用。

### 5.2 键盘交互

候选列表打开时，以下按键优先作用于候选列表，而不是普通输入行为：

- `ArrowDown`：下移一项；到底后循环到首项。
- `ArrowUp`：上移一项；到顶后循环到末项。
- `Home`：跳到首项。
- `End`：跳到末项。
- `Tab`：切换到下一项。
- `Shift+Tab`：切换到上一项。
- `Enter`：插入当前高亮项。
- `Escape`：关闭当前这一轮候选列表。

补充说明：

- 候选列表打开时，`Enter` 不会提交消息，而是优先用于选中候选。
- 候选列表打开时，`Shift+Enter` 也不会换行，而是同样优先用于选中候选。
- `Escape` 关闭的是当前查询态下的候选；继续编辑查询词后，候选可以再次出现。

## 6. 插入后的表现

### 6.1 可见形式

选中候选后，输入框内不会保留原始 `@查询词`，而是直接替换成一个内联引用 token。

该 token 的可见文本仅为：

- 文件名，或
- 文件夹名。

不会在 token 内直接展示完整路径。

例如：

- 选择 `src/components/app-titlebar.tsx` 后，输入框里显示 `app-titlebar.tsx`。
- 选择 `src` 文件夹后，输入框里显示 `src`。

### 6.2 插入范围

插入时，仅替换当前 `@查询词` 这一段文本，不会额外插入空格，也不会主动修正前后标点。

例如：

- 输入框内容为 `Review @app-ti`，插入后可见内容为 `Review app-titlebar.tsx`。
- 输入框内容为 `看下 @read中文`，插入后可见内容为 `看下 README.md中文`。

即：

- 前后已有字符会被原样保留。
- 中英文之间不会自动补空格。
- 标点附近不会自动补空格。

### 6.3 光标位置

插入完成后，光标落在该 token 之后。

## 7. token 的编辑规则

### 7.1 原子性

已插入的引用 token 是原子对象，不能像普通文本那样在中间逐字符编辑。

用户可感知行为：

- 无法把 token 当作普通字符串拆开修改。
- 光标不会稳定停留在 token 内部字符之间。
- 一旦编辑动作触及 token，系统会按整个 token 处理。

### 7.2 删除行为

当光标与 token 相邻时：

- 在 token 右侧按 `Backspace`，会删除整个 token。
- 在 token 左侧按 `Delete`，会删除整个 token。

### 7.3 范围删除与替换

当选区与 token 有交叉时，系统会先把选区扩展到 token 的完整边界，再执行后续操作。

因此：

- 删除一段跨过 token 的选区时，token 会整块删除。
- 剪切一段跨过 token 的选区时，token 会整块剪切。
- 用粘贴内容覆盖一段跨过 token 的选区时，token 会整块被替换。
- 不会保留“半个 token”。

### 7.4 普通文本编辑对 token 的影响

在 token 前后继续输入、删除或换行时：

- 未被触及的 token 会保留。
- 其引用关系保持有效。
- 后续 token 的位置会随正文偏移自动调整。

## 8. 复制、剪切、粘贴

### 8.1 复制

复制时，引用 token 不按可见文本复制，而是按“可发送文本”复制。

复制结果中的引用格式为：

```md
[显示名称](工作区相对路径)
```

例如：

- 输入框中可见为 `README.md` 的 token，复制到剪贴板后为 `[README.md](README.md)`。
- 输入框中可见为 `app-titlebar.tsx` 的 token，复制到剪贴板后为 `[app-titlebar.tsx](src/components/app-titlebar.tsx)`。

如果复制的是混合内容，则结果为“普通文本 + 序列化引用”的拼接文本。

### 8.2 剪切

剪切行为与复制一致：

- 先以序列化格式写入剪贴板。
- 再从输入框中删除所选内容。

若选区触及 token，则按整个 token 处理。

### 8.3 粘贴

粘贴仅按纯文本插入。

这意味着：

- 粘贴普通文本时，按普通文本处理。
- 粘贴形如 `[README.md](README.md)` 的文本时，也只会作为普通文本插入。
- 粘贴不会自动把 Markdown 引用重新识别为 token。

## 9. 发送给 Agent 时的结果

### 9.1 序列化规则

输入框内部的 token 在发送前会统一转换为文本引用格式：

```md
[显示名称](工作区相对路径)
```

其中：

- `显示名称` 为 token 在输入框中的可见名称。
- `工作区相对路径` 永远是相对工作区根目录的路径，不是绝对路径。

### 9.2 发送时机与清空

- 按 `Enter` 可发送消息。
- 按 `Shift+Enter` 可插入换行，但前提是候选列表未打开。
- 发送成功后，输入框内容会清空，已插入的引用 token 也随之清空。

### 9.3 发送限制

以下情况不会发送：

- 未打开工作区。
- 发送内容在序列化后去除首尾空白仍为空。

## 10. 显示与发送的差异

该功能存在“输入框显示内容”与“实际发送内容”两层表达：

- 输入框内显示的是简化后的名称 token，便于阅读和继续编辑。
- 实际发送给 Agent 的是包含相对路径的引用文本，便于精确定位。

示例 1：

- 输入框显示：`Review app-titlebar.tsx`
- 实际发送：`Review [app-titlebar.tsx](src/components/app-titlebar.tsx)`

示例 2：

- 输入框显示：`看下 README.md中文`
- 实际发送：`看下 [README.md](README.md)中文`

## 11. 同名项与路径歧义

当前实现下，同名文件或文件夹可以同时出现在候选列表中。

区分方式如下：

- 候选列表内通过“名称 + 父级相对路径”区分。
- 插入到输入框后，token 只显示名称，不显示父级路径。
- 真正发送时，通过相对路径区分具体引用对象。

因此：

- 两个不同目录下的同名文件，插入后在输入框内可能看起来完全相同。
- 但发送给 Agent 的内容仍能区分，因为相对路径不同。

## 12. 输入法与焦点相关行为

### 12.1 输入法组合输入

在输入法组合输入过程中：

- 输入框不会立即按普通输入节奏提交中间态内容。
- `@` 检索相关更新以最终上屏内容为准。

对中文输入的直接影响是：

- 正在拼字或候选上屏过程中，不以中间态文本驱动最终引用更新。
- 组合输入结束后，再按最终文本继续判定是否进入 `@` 检索态。

### 12.2 失焦

输入框失焦后：

- 候选列表关闭。
- 当前已插入 token 保留，不会丢失。

## 13. 当前实现结论汇总

该功能当前的产品语义可以概括为：

- `@` 用于引用“当前工作区中的文件或文件夹”。
- 引用对象在输入框中以“名称 token”显示。
- 引用对象在发送给 Agent 时以“名称 + 相对路径”文本发送。
- token 是原子对象，可整体删除、剪切、复制，但不能当普通文本拆开编辑。
- 粘贴不会把序列化引用恢复成 token。
- 搜索支持文件名、路径、片段、多词与中英文混输。
- 候选列表优先服务于选择引用，打开时会接管 `Enter`、`Tab`、方向键等关键操作。
